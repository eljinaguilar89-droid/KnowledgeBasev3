Technical Specification - KnowledgeBasev3

1) Project Overview
This repository implements a knowledge base web application (KnowledgeBasev3) with a React + TypeScript frontend and an Express + Node backend. It supports user auth, role-based access, article/category management, logging, and an API for public integrations. The app uses PostgreSQL via Drizzle ORM when a database is configured; otherwise it falls back to in-memory data for development.

2) Tech Stack
- Frontend: React 19 + TypeScript, built with Vite
- Styling: Tailwind CSS (+ @tailwindcss/typography)
- State / Context: React Context (AuthContext)
- Charts / UI libs: Recharts, lucide-react, motion
- Backend: Node.js + Express
- Database: PostgreSQL (drizzle-orm + drizzle-kit)
- DB driver: `pg` (node-postgres)
- Auth / Security: `bcryptjs` for password hashing; API key support
- Dev tooling: tsx for running TypeScript server in dev, Vite middleware for front-end HMR
- Containerization: Dockerfile + docker-compose.yml (Postgres service + app)
- Environment: dotenv (.env.local)

4) Architecture & Components
- Frontend (src/)
  - `AuthContext.tsx`: manages login state in `localStorage` under the key `ied_user`.
  - `App.tsx`: top-level view routing and data fetch logic (fetches `/api/articles` and `/api/categories` when logged in).
  - Views and components under `src/views` and `src/components` implement the UI (Dashboard, Editor, Browse, Analytics, Users, Logs, etc.).
  - `AiChatDrawer` and `ApiIntegrationView`: client-side integrations (project includes `@google/genai` dependency for AI usage).

- Backend (server.ts)
  - Express server exposing REST endpoints under `/api`.
  - Dev: Vite is started in middleware mode so the same process serves the frontend during development.
  - Production: the `dist/` directory is served as static assets; Express returns `index.html` for SPA routes.
  - Routes include (non-exhaustive):
    - Auth: `POST /api/auth/signup`, `POST /api/auth/signin`, `POST /api/auth/generate-key`
    - Articles CRUD: `GET/POST/PUT/DELETE /api/articles`
    - Categories: `GET/POST /api/categories`
    - Users: `GET /api/users`, `PUT /api/users/:id/role`, `DELETE /api/users/:id`
    - System logs: `GET/POST /api/logs`
    - Public API namespace: `/api/public/v1/*` (requires API key when DB enabled)

- Database (src/db)
  - Uses Drizzle ORM (Postgres adapter). Configuration is in `drizzle.config.ts`.
  - `src/db/schema.ts` defines tables: `users`, `categories`, `system_logs`, `articles`.
  - `src/db/index.ts` creates a `pg.Pool` and the Drizzle client, with optional SSL handling for cloud-hosted Postgres.
  - `drizzle/` contains SQL snapshots and migration output.

5) Data Model (high level, from schema.ts)
- `users`: id, email (unique), password (hashed), name, role, api_key, created_at
- `categories`: id, title, filter_category, icon, count, color_class, group
- `system_logs`: id, level, message, date (YYYY-MM-DD), created_at
- `articles`: id, title, excerpt, content, author, date, views, category, categoryColor, badge, status, accessLevel, attachments (JSON), created_at

6) Authentication & Authorization
- Passwords are hashed with `bcryptjs` on signup (`/api/auth/signup`).
- Sign-in (`/api/auth/signin`) validates credentials and returns user profile + `apiKey` if present.
- API Key flow: `/api/auth/generate-key` issues keys prefixed with `ied_...` and stored in `users.api_key` when DB enabled.
- Public integration routes under `/api/public/v1/*` check `x-api-key` header or `apiKey` query param against `users.api_key` when a database is configured.
- Frontend role-based UI logic in `App.tsx` filters article visibility by `user.role` and `article.accessLevel`.

7) Logging & Telemetry
- The server supports storing logs in the `system_logs` table or in-memory logs when `DATABASE_URL` is not set.
- Endpoints: `GET /api/logs?date=YYYY-MM-DD` and `POST /api/logs` to retrieve and create logs.

12) Known Implementation Details & Tips
- The server falls back to in-memory collections (`mockArticles`, `mockCategories`, `inMemoryLogs`) when `DATABASE_URL` is not set â€” convenient for local dev without Postgres.
- The frontend stores the logged-in user in `localStorage` (`ied_user`). Clearing it forces re-authentication.
- Role-based visibility logic is implemented client-side in `App.tsx` (additional server-side checks are minimal); consider adding server-side authorization checks if stronger enforcement is required for production.
